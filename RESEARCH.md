# Agent-Led Input/GT Discovery Research (PyTorch First)

This is a living research document. Update `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` as we iterate across conversations.

This document is intentionally separate from production implementation planning in `PLAN.md`. The purpose here is to validate ideas quickly with temporary research scripts and collaborative analysis before deciding what belongs in Concierge production code.

## Purpose / Big Picture

We want a new way to identify model inputs and ground truths for Tensorleap integration authoring. The current static-first method is useful for model-file discovery, but not strong enough for reliable input/GT recommendation. The research goal is to prove a better method:

1. Use static search only to generate leads.
2. Use an agent to semantically read repository code from those leads.
3. Present proposed input/GT mappings to the user for confirmation.
4. Use runtime/model introspection only as a secondary fallback or validation layer.

If this works, we will have a practical, evidence-based path for recommending correct input encoders and ground-truth encoders in repositories without existing Tensorleap integration.

## Non-Production Research Constraints (Hard Rules)

1. Research mode only until explicitly changed by user.
2. No production Concierge code changes during this phase.
3. Temporary scripts and artifacts are allowed for experimentation only.
4. No CI integration for research scripts.
5. No TDD requirement for research scripts.
6. Keep experiments lightweight, reversible, and fast to rerun.
7. Work is collaborative: decisions, prompt changes, and heuristic changes are reviewed with the user before being treated as direction.

## Scope

Initial scope is intentionally restricted to PyTorch repositories.

In scope:

1. Lead discovery using PyTorch-oriented static signals (`DataLoader`, `Dataset`, train/val flow).
2. Agent-driven semantic analysis to infer candidate inputs and GTs.
3. Comparison against known fixture post states as ground truth.
4. Iterative prompt/heuristic refinement based on measured quality.

Out of scope for this phase:

1. TensorFlow-specific detection.
2. Production wiring in Concierge.
3. CI policy, release process, and hardening.
4. End-to-end runtime execution as primary discovery path.

## Research Hypotheses

1. PyTorch lead signals are sufficient to direct the agent to relevant data semantics quickly.
2. Train/validation entry points are high-value anchors for discovering input/GT mapping.
3. Agent semantic traversal from leads (imports, call chains, dataset classes, collate and batch unpacking) can recover usable input/GT candidates.
4. User confirmation after agent proposal reduces wrong authoring decisions.
5. Runtime model inspection is most useful as fallback/cross-check, not as primary discovery.

## Candidate Lead Signals (PyTorch)

Primary signals:

1. `torch.utils.data.DataLoader` import/instantiation.
2. `Dataset` implementations and custom dataset classes.
3. Training loop entry points (for example scripts/functions named `train`, `fit`, `main`).
4. Validation loop entry points (`val`, `validate`, `evaluation`, `test` in training context).

Secondary signals:

1. Batch unpacking patterns (`for batch in ...`, tuple/dict unpacking).
2. `collate_fn` definitions and references.
3. Data transform chains near dataset construction.
4. Model forward-call sites in train/val loops.

## Agent Task Definition (Research Version)

Input detection is treated as an agent task, not a static-only parser task.

Research agent contract:

1. Read-only investigation of repository code.
2. Start from lead pack generated by static search.
3. Trace code paths from training/validation entry points to data loading and batching.
4. Infer candidate model inputs and candidate ground truths.
5. Return structured output with evidence and confidence.
6. Explicitly list unknowns and ambiguities.

Expected output fields (draft):

1. Candidate inputs (name, shape/type hints, evidence locations).
2. Candidate ground truths (name, semantics, evidence locations).
3. Proposed encoder mapping (input encoders + GT encoders).
4. Confidence per candidate.
5. Unresolved questions for user confirmation.

## Method Overview

Phase A: Build lead extraction and context packaging.

1. Write temporary Python scripts that scan repo text for PyTorch lead signals.
2. Produce a compact lead pack (files, symbols, snippets, and ranks).
3. Keep scripts under a dedicated temporary research area.

Phase B: Agent semantic inference.

1. Define a static system prompt for read-only semantic investigation.
2. Feed lead pack + targeted repository context to the agent.
3. Collect structured candidate input/GT mapping output.

Phase C: Ground-truth evaluation.

1. Compare agent output to fixture post-state integration definitions.
2. Score matches/misses for inputs and GTs.
3. Capture failure modes and uncertainty patterns.

Phase D: Refinement loop.

1. Refine lead heuristics and prompt instructions.
2. Re-run experiments on the same fixtures.
3. Track whether precision/recall and evidence quality improve.

Phase E: Runtime fallback exploration (secondary).

1. Evaluate lightweight runtime/model introspection only after code-first approach is characterized.
2. Use it as cross-check/fallback hints, not as first-line discovery.

## Surprises & Discoveries

- Observation: Static model-file discovery is not enough to infer semantic input/GT mapping in customer code.
  Evidence: Discussion with Tensorleap developer plus current Concierge limitations.
- Observation: `DataLoader` references and train/val entry units are likely strong discovery anchors for PyTorch.
  Evidence: Developer-guided workflow description and repository structure expectations.
- Observation: Runtime introspection is feasible in Concierge context but should be secondary at first.
  Evidence: Existing preprocess/integration-test flow already executes repository/runtime logic.

## Outcomes & Retrospective

Current status: direction is defined and converted into a concrete, collaborative research plan. No execution results yet.

Known risks:

1. Agent may overfit to naming conventions and miss non-standard pipelines.
2. Evidence extraction might be noisy in large repositories.
3. Ground-truth mapping from fixture integration files may require interpretation choices.

Mitigations:

1. Require evidence-backed outputs and explicit unknowns.
2. Use staged difficulty progression (`webinar` -> `yolov5_visdrone` -> `ultralytics`).
3. Record scoring rubric decisions in this file to keep comparisons consistent.

## Idempotence and Recovery

1. Research scripts should be rerunnable without mutating fixture source repos.
2. Each experiment output should be written to a timestamped result path.
3. If an experiment design proves weak, keep artifacts and record why before replacing it.
